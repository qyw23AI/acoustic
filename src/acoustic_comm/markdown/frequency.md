Thought for 1m 12s

不是说你这组 `1500 / 2500 Hz` 有问题。我的意思是：

**在“缩短时长”这件事上，频率几乎不是主矛盾。**

时长主要由这些决定：

* `frame_bits`
* `bit_dur`
* `repeat_n`
* `chirp_dur`
* 各段静音

把 `f0` 从 1500 改成 1800，或者把 `f1` 从 2500 改成 3200，**总时长一秒都不会少**。所以在还没上设备之前，我会先动协议结构，不会先纠结频点。

### 那 `1500 / 2500` 到底怎么样？

就纸面上看，**这组并不差**：

* 两个频点间隔 `1000 Hz`，分离度够大，做 2-FSK 比较友好
* 在你现在的 `bit_dur=0.05` 下：

* `1500 * 0.05 = 75` 个周期
* `2500 * 0.05 = 125` 个周期

都是整数周期，做能量检测/Goertzel/相关检测都比较顺
* 以后如果你把 `bit_dur` 压到 `0.02` 或 `0.03`，这组频率其实也还挺顺：

* `0.02s` 时分别是 `30 / 50` 个周期
* `0.03s` 时分别是 `45 / 75` 个周期

所以我不会把这组频率定义成“有问题”。

### 我为什么说它要放到后面再优化？

因为**频率的优劣非常依赖真实硬件**，纸上推不出来。

同一组频点，最终效果会被这些东西决定：

* 喇叭在那个频段到底响不响
* 麦克风在那个频段到底灵不灵
* 机器人电机、齿轮、风扇噪声主要落在哪段
* 场地混响会不会把某些频段放大/削弱
* 设备本身有没有频偏、采样偏差

你文档里其实也已经写到了两件关键事：

1. **f0/f1 最终必须靠实测决定**，因为设备可能不支持更高频段。
2. **频偏/采样率偏差会明显影响检测**，接收端放宽测量容差后成功率会提高很多。

所以我前面那句话真正想表达的是：

> 在没上设备前，`1500 / 2500` 完全可以先当默认值；
> 
> 等你把协议缩短以后，再用实测去决定要不要换频点。

### 不走超声，完全没问题

这点我同意你。

文档只是建议“如果硬件支持，可以往更高频试”，不是说必须上 18–20kHz。它同时也明确提醒了：喇叭/麦克风可能根本不支持，必须实测。

如果不考虑超声，我会这样定策略：

**先把搜索范围锁在可听高频但不太高的区间**，比如大致 `1.5kHz ~ 4kHz` 或 `2kHz ~ 4.5kHz`。

这通常比特别低的频段更容易避开一些机械噪声，但又不会像高频那样太吃硬件响应。

### 如果是我，没实测前会怎么做

我会直接这样处理：

**第一步：先不改 `1500 / 2500`**

* 因为它能用来推进协议开发
* 而且它和 `20ms/30ms/50ms` 这些 bit 时长的配合都还不错

**第二步：先优化真正影响延迟的东西**

* 去掉快链路里的 `crc`
* 去掉 `seq`
* 从 `24bit` 改成直接发 `8bit codeword`
* `repeat_n` 从 3 降到 2，甚至首帧有效就动作

**第三步：上设备后再扫 4 组频点**

比如先测这几组：

* `1500 / 2500`
* `1800 / 3000`
* `2200 / 3400`
* `2600 / 4000`

然后在真实条件下比较：

* 漏检率
* 误检率
* 电机开着时的成功率
* 距离变化后的成功率

最后再锁一组。

### 一句话结论

**你的 `1500 / 2500 Hz` 现在没什么明显毛病。**

我之所以不建议先动它，不是因为它错了，而是因为：

* 它**不解决时长问题**
* 它的优劣**高度依赖设备实测**
* 你当前最该优化的是**协议结构，不是频点**
