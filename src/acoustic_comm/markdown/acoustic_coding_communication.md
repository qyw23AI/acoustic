# 声学编码通信

## 基本定位

本方案用于 R1 与 R2 之间的辅助通信。

- 主通信方式为 AprilTag 识别
- 声学编码通信作为辅助链路使用
- 声学链路主要承担以下任务：
  1. 视觉链路正常时，作为辅助触发与兜底
  2. 视觉短时不稳定时，承担少量关键命令下发
  3. 在保证基本可靠性的前提下，尽量缩短 wav 时长，提高机器人实时反应速度

因此，本方案不追求“所有命令都走同一套最稳协议”，而是将命令分为**快动作链路**和**慢动作链路**。

---

## 基本思路

1. 列出 R2 所有需要 R1 发送指令的行为，然后将其与编码一一对应。
2. 将编码转变为 bit 序列，0 对应 `f0`，1 对应 `f1`，这样每条指令都有一个对应的音频。
3. 音频由 R1 的喇叭播放，R2 的麦克风收音，实现 R1 与 R2 之间的非无线通信。
4. 对时延敏感的命令走快动作链路；误触代价高、需要更稳处理的命令走慢动作链路。

---

## 命令表

| 区域 | 行为 | CMD_ID (hex) | 二进制 | 链路 |
|---|---|---:|---|---|
| 武馆 | R2 在端头架等待，收到指令后开始拼接 | `0x8A` | `10001010` | 快 |
| 武馆 | R2 拼接结束后，用人的肉眼判断是否拼接成功，随后进行拿下一个端头的动作 | `0x5C` | `01011100` | 慢 |
| 武馆 | R2 接到 R1 离开武馆的指令，在 R1 离开武馆后进入梅林 | `0x27` | `00100111` | 快 |
| 梅林 | 前三个梅林方块放有只能由 R1 拾取的 KFS，R1 完全拿走后，R2 再上梅林 | `0xE1` | `11100001` | 快 |
| 梅林 | 人的肉眼判断 R2 已经拿够了 KFS，R1 进入对抗区后，R2 也跟着离开梅林 | `0x72` | `01110010` | 快 |
| 对抗区 | R2 站到 R1 上将 KFS 放在九宫格顶层 | `0x39` | `00111001` | 慢 |
| 全场 | R2 需要重试，由 R1 给出指令 | `0x95` | `10010101` | 慢 |
| 全场 | STOP / ABORT：立刻停下当前由声学触发的动作，进入等待状态 | `0xC6` | `11000110` | 慢 |

### 说明

- 快动作链路用于对时延敏感、需要尽快响应的动作。
- 慢动作链路用于误触代价高、需要更稳处理的动作。
- STOP / ABORT 为最高优先级安全命令，建议在任意阶段都可接收。

## CMD_ID 选取原则

CMD_ID 需要具有识别性。选取原则如下：

1. 任意两条指令的 8bit 编码，最好至少有 3 位以上不同
2. 编码尽量 0/1 均衡
3. 编码尽量有翻转，这样更不容易受影响

### 当前采用与预留的码字池

| 推荐序号 | CMD_ID | 二进制 | 说明 | 当前用途 |
|---:|---:|---|---|---|
| 1 | `0x8A` | `10001010` | 基础核心码字（均衡、翻转多） | 已用 |
| 2 | `0x5C` | `01011100` | 与 #1 汉明距 = 5（≥5） | 已用 |
| 3 | `0x27` | `00100111` | 与 #1 = 5、与 #2 = 6（前三条两两 ≥ 5） | 已用 |
| 4 | `0xE1` | `11100001` | 加入后“前4条两两 ≥ 4”仍成立 | 已用 |
| 5 | `0x72` | `01110010` | 加入后“前5条两两 ≥ 4”仍成立 | 已用 |
| 6 | `0x39` | `00111001` | 加入后“前6条两两 ≥ 4”仍成立 | 已用 |
| 7 | `0x95` | `10010101` | 加入后“前7条两两 ≥ 4”仍成立（此池上限） | 已用 |
| 8 | `0xC6` | `11000110` | 从这里起：无法再保持全局 ≥ 4，降为“全局 ≥ 3” | STOP |
| 9 | `0x4B` | `01001011` | 0/1 很均衡、翻转多；与现有集合整体区分度较好 | 预留 |
| 10 | `0xCD` | `11001101` | 区分度不错（但 1 略多 = 5） | 预留 |
| 11 | `0x57` | `01010111` | 均衡、翻转多（但会进一步引入一些 3 距离对） | 预留 |
| 12 | `0xA4` | `10100100` | 均衡、结构清晰 | 预留 |
| 13 | `0x68` | `01101000` | 均衡、与部分码字差异较大但也会出现 3 距离对 | 预留 |
| 14 | `0xB3` | `10110011` | 1 的数量多一些（5），靠后放 | 预留 |
| 15 | `0xD0` | `11010000` | 1 的数量少一些（3），翻转相对少，放最后 | 预留 |

## 音频处理

1. 采样率 `16000`。每秒采 `16000` 个点，人声主要信息大概在 `0～8kHz` 以内，16k 采样率能覆盖到 8kHz。
2. 使用正弦波做 FSK 提示声音。正弦波是最纯的单频声音，只有一个频率成分。
3. `bit_dur = 0.05`，每个 bit 的持续时间为 `50ms`。
4. 给每个哔加 `5～10ms` 的淡入淡出，不然哔音和静音硬拼接时，波形可能从某个非零值瞬间跳到 0，会引入很宽频的能量。
5. 当前工程默认使用：
   - `f0 = 1500Hz`
   - `f1 = 2500Hz`
6. 当前工程默认使用 chirp 同步头：
   - `CHIRP_F0 = 1200Hz`
   - `CHIRP_F1 = 3200Hz`

### 说明

- 当前阶段不再压缩 `bit_dur`，统一保持 `0.05s`
- 后续优化时长主要通过以下方式完成：
  - 减少 payload bit 数
  - 减少重复次数
  - 缩短 chirp
  - 调整静音结构
- 当前默认不采用接近超声的频段作为比赛配置；是否上移频点，由后续设备实测决定

## 当前采用的两类发射结构

## 一、快动作链路

### 适用命令
- `0x8A`
- `0x27`
- `0xE1`
- `0x72`

### 设计目标
- 尽量缩短 wav 时长
- 提高机器人实时反应速度
- 依赖主系统进行额外约束，不追求最重校验

### 帧结构

```text
[silence_head] + [chirp] + [cmd]
```

### 重复结构

```text
one_frame + gap + one_frame
```

### 参数

* `cmd_bits = 8`
* `seq_bits = 0`
* `crc_bits = 0`
* `use_crc = False`
* `repeat_n = 2`
* `silence_head_s = 0.01`
* `silence_tail_s = 0.00`
* `gap_s = 0.02`
* `chirp_dur = 0.06`

### 时长估算

单帧：

* `0.01 + 0.06 + 8 * 0.05 = 0.47s`

整条 wav：

* `0.47 + 0.02 + 0.47 = 0.96s`

### 说明

* 快动作链路只发 `cmd`
* 不再使用 `seq`
* 不使用 `crc`
* 发送两次，主要用于提高命中率，而不是做重型投票

---

## 二、慢动作链路

### 适用命令

* `0x5C`
* `0x39`
* `0x95`
* `0xC6`

### 设计目标

* 提高关键动作可靠性
* 降低误触代价
* 保留基础错误检测能力

### 帧结构

```
[silence_head] + [chirp] + [cmd + crc]
```

### 重复结构

```
one_frame + gap + one_frame
```

### 参数

* `cmd_bits = 8`
* `seq_bits = 0`
* `crc_bits = 8`
* `use_crc = True`
* `repeat_n = 2`
* `silence_head_s = 0.02`
* `silence_tail_s = 0.00`
* `gap_s = 0.06`
* `chirp_dur = 0.08`

### 时长估算

单帧：

* `0.02 + 0.08 + 16 * 0.05 = 0.90s`

整条 wav：

* `0.90 + 0.06 + 0.90 = 1.86s`

### 说明

* 慢动作链路发 `cmd + crc`
* 不再使用 `seq`
* 与原来的 `cmd + seq + crc + 3次投票` 相比，时长显著缩短，同时保留基础错误检测能力

---

## 静音结构

原始长帧结构通常写成：

```
[silence_head] + [chirp] + [payload] + [silence_tail]
```

当前方案改为：

```
[silence_head] + [chirp] + [payload]
```

并保留帧间 `gap`。

### 当前约定

1. **保留 `silence_head`**

* 用于给 chirp 一个干净起点
* 降低帧起始边界过硬的问题
2. **去掉 `silence_tail`**

* chirp 和 payload 都进行 fade 后，帧尾可以直接收束到接近 0
* 不再额外保留尾静音
3. **保留 `gap`**

* 用于分隔相邻重复帧
* 避免两帧粘连
* 提高接收端对下一帧起点的识别能力

## 防干扰机制

## 1\. preamble（备选简化同步方案）

可以使用前导码（preamble），即一段已知的固定序列，用于接收端做检测 / 同步。

preamble 提前告知接收端 bit 的频率和边界。

例如：

* 使用交替序列作为前导
* 在中低噪声下效果尚可
* 当信噪比下降后，payload 容易与期望不一致

该方案可以作为进一步压缩 wav 时长的备选思路，但当前默认不作为比赛主方案。

---

## 2\. chirp（当前默认同步方案）

chirp 是比普通 preamble 更强的同步头。

其扫频在时间上有独特结构，模板匹配后会出现明显峰值。

### 当前默认 chirp 参数

* `CHIRP_F0 = 1200`
* `CHIRP_F1 = 3200`

### 当前工程约定

* 发送端 chirp 需要加 fade
* 接收端 chirp 模板需要与发送 chirp 保持一致，包括 fade
* 快动作与慢动作使用不同的 chirp 时长

---

## 3\. CRC（慢动作链路使用）

CRC（循环冗余校验，Cyclic Redundancy Check）用于检测错误。

### 优点

* 几乎不存在误通过的情况
* 对随机错误有很强的检测能力

### 缺点

* CRC 只能检测错误，不能纠正错误
* 如果采用 `cmd + seq + crc` 再配合多次投票，wav 会明显变长，实时性差

### 当前采用方式

* 快动作链路不使用 CRC
* 慢动作链路使用 `cmd + crc`
* 不再要求操作手维护 `seq`

---

## 为什么不再使用原始长链路作为默认方案

原始长链路思路是：

```
[silence] + [chirp] + [cmd + seq + crc] + [silence]
```

并配合：

* `repeat_n = 3`
* 帧间静音
* 接收端投票

这样总 wav 会达到 4 秒级，不适合作为比赛默认链路。

### 结论

1. `CRC + 3次投票` 更适合重大动作和实验环境验证
2. 对于比赛中需要快速完成的动作，时长过长，实时性差
3. 纯靠缩短 `bit_dur` 来压 wav 风险较大；已有结果表明，将 `bit_dur` 从 `0.05s` 压到 `0.04s` 后，虽然 wav 变短，但校验失败
4. 因此当前方案改为：

* 快动作：`cmd`
* 慢动作：`cmd + crc`
* 全部不再依赖操作手维护 `seq`

---

## 无 seq 条件下的重复触发保护

弃用 `seq` 的原因是：

* 不希望比赛时让操作手维护序号
* 简化操作流程
* 缩短音频时长

但弃用 `seq` 后，需要额外的重复触发保护。

### 建议做法

1. **cooldown 机制**

* 同一类命令在短时间内不重复触发
2. **上层状态约束**

* 主系统结合 AprilTag 和任务状态，决定某条命令当前是否允许执行
3. **STOP / ABORT 兜底**

* 一旦误触，由操作手立即发送 STOP 指令，使机器人停下当前由声学触发的动作

---

## 接收端处理原则

1. 先检测 chirp 峰值
2. 根据 chirp 起点定位 payload
3. 按 bit 窗口逐段解码
4. 对慢动作链路执行 CRC 校验
5. 对重复帧进行简单重复确认
6. 对通过校验的结果做 cooldown / 上层逻辑约束
7. STOP / ABORT 为最高优先级安全命令，成立后立即停下当前由声学触发的动作

## 其他测试结果

## 1\. 白噪声

加入白噪声后，同样是 `SNR = -15dB` 后开始下降。

此时噪音幅度大概是信号的 `5.6` 倍。

## 2\. 粉噪声

加入粉噪声后，结果与白噪声类似。

## 3\. 频率偏移 / 采样率偏差

赛场喧闹和收音效果不一定好，所以必须考虑频率偏移或采样率偏差。

### 现象

* 只在频率完全没有偏差的时候，成功率最高
* 稍微偏差一点，就可能检测不到

### 处理

* 增加接收端测量容差后，成功率会明显提高

### 结论

* 接收端必须考虑频偏与采样偏差
* 模板匹配和 bit 判决都要保留一定容差
* 不能只按理想频率去设计比赛参数

---

## 实测部分建议

1. 当前默认先不采用接近超声的频段作为比赛方案。

是否上移频点，必须结合喇叭 / 麦克风的实际频响做实测决定。
2. 硬件建议：

* 使用定向喇叭 / 定向麦克风
* 或者喇叭加简单号角 / 导音管
* 麦克风加短声管
* 减少机器人自身电机噪声直灌麦克风
* 降低空气通信的随机性
3. 后续实测重点：

* 喇叭 / 麦克风频响
* 距离变化
* 电机噪声影响
* 场地混响
* 误检率与漏检率
* STOP 命令在实战中的响应速度与可靠性

## 当前方案总结

本方案的核心思路如下：

1. 命令仍然使用固定 8bit `CMD_ID`
2. 命令分为快动作链路和慢动作链路
3. 快动作链路只发 `cmd`
4. 慢动作链路发 `cmd + crc`
5. 不再使用 `seq`
6. 不让操作手承担维护序号的压力
7. 使用 STOP / ABORT 作为安全兜底命令
8. 保持 `bit_dur = 0.05`
9. 缩短 wav 的主要手段为：

* 减少 payload bit 数
* 减少重复次数
* 缩短 chirp
* 调整静音结构
10. 声学链路始终作为辅助通信手段，与 AprilTag 主链路配合使用
