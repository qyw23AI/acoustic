可以。你这次“明显变好”，不是靠某一个魔法参数，而是**接收链路被同时修了三层**：

第一层，先把“根本听不对”的输入链路修正了。

你的协议域还是 `sr=16000`，但监听器新增了 `input_sr`，允许麦克风先按设备真正支持的采样率采集，再重采样回协议采样率；代码里就是 `input_sr` + `_resample_block(...)` 这一套。你给机器人麦最终用的是 `input_sr=48000`，而协议/解码仍然保持在 16k 域里。这样做解决的不是“判决”问题，而是更基础的“麦克风数据先得正确进来”问题。之前设备对 16k 直开不友好时，后面 fast/slow 再怎么调都不稳；改成 `48k 采集 → 16k 解码` 后，检测链路才真正站住了。

第二层，**双触发**主要是靠“只看新峰 + 接受后清空 + 一段时间 suppress”一起压下来的。

README 里已经把这套策略写得很明确：实时监听器现在是 **stateful peak scan**，只处理新的 chirp 峰值；接受一次触发后会 **clear ring after accept**，把 ring buffer 清掉；然后再进入一段 **post-accept holdoff**，Fast 默认约 `1.25s`，Slow `2.0s`，STOP `3.0s`。这三件事叠在一起，等于把“同一条命令的尾波、回响、重复扫到旧峰”都堵住了。代码里对应的是 `ModeScanState.last_peak_abs` 记住已经扫到哪里，`clear_ring_after_accept=True`，以及 `fast_post_accept_s / slow_post_accept_s / stop_post_accept_s` 这些参数。也就是说，之前你明明只播一次却会连触发两次，本质上是同一包被重复当成新事件；现在这条路被封住了。

第三层，**漏检少了**，主要是因为“放宽接受条件”从错误的位置，移到了正确的位置。

你之前提到“投票不通过，一次也检测不到”，而现在 `fast_accept_min=1`、`slow_accept_min=1`，只有 STOP 仍然要求 2 帧确认。代码里的 `_summarize_ok_frames()` 和 `_required_accept_min()` 也是这个逻辑：普通 Fast / Slow 只要 1 个有效帧就能过，STOP 才更严格。这样做的效果很直接：以前如果现场只能稳定抓到重复帧里的 1 帧，旧策略会整包丢掉；现在单帧也能接受，所以漏检会明显下降。

但它不是“无脑放宽”。

为了避免把误检也一并放大，Fast 侧又额外加了一个**按预期重复窗口配对**的约束。README 里写的是“Fast 侧额外使用预期重复窗口做配对，减少错误重复峰带来的误触发”；代码里 `_is_expected_fast_repeat(...)` 会检查第二个峰是不是落在“第一帧之后、理论上重复帧该出现的位置附近”，并用 `fast_repeat_tol_s` 给容差。这样你得到的是一种更像“有物理结构约束的放宽”：允许单帧接受，所以漏检少；但如果要把两帧视为同一包，它们必须在合理时间关系里，所以误触发和乱配对又被压住。

第四层，解码本身也更抗“边界没切准”了。

在 `_decode_frame_near_peak()` 里，代码不是拿粗峰值直接硬解，而是先在附近做一次本地 refine，找到更好的 chirp 起点；然后还会尝试多个 bit-phase 偏移。简单说，就是“先把帧头位置再对准一点，再试几种 bit 切窗相位”。现场里只要 chirp 起点有一点偏、或者 payload 的 bit 边界没刚好落在最理想的位置，老版本就容易出现“明明听到了，但帧校验过不了”；现在多了一层局部细化和多相位尝试，能把很多原本会丢掉的帧捞回来，所以漏检进一步下降。

至于你说“**音量可以调很大，不怕削波**”，这里要分开看。

代码里确实加入了 `clip_warn_level` 和 `mic_level` 的监控，debug 会提示输入峰值和潜在 clipping，这能帮助你发现问题、避免误判来源；README 也明确提到调试输出会提示 clipping。也就是说，系统现在**更会识别和暴露削波风险**了。

但更关键的是：你看到“音量拉大后还是不容易坏”，并不只是因为“加了个剪裁告警”，而是因为前面那几层改动把系统从“很脆”变成了“有冗余”：

* 输入链路正确了，麦克风拿到的是稳定、可解的信号。
* chirp 检测和 bit 切相位更宽容了，小失真不至于直接整包报废。
* 接受策略不再苛求两帧都完美，因此偶发失真不会让整次命令归零。
* 触发后会清 ring 并 suppress，所以大音量带来的尾波和混响，也不容易变成第二次假触发。

所以你实测感受到的是：“音量大了也没那么容易崩”。

更准确地说，是**系统对失真、尾波、边界误差的容忍度提高了**，而不是“削波从物理上消失了”。

一句话总结这次为什么会成功：

**不是单纯把阈值调低了，而是把实时监听器改成了“输入采样率适配 + 状态化只扫新峰 + 接受后清空与抑制 + Fast 结构化配对 + 局部 refine/多相位解码”的组合。**

于是它同时解决了三件事：

同一包不再被反复认成新命令，所以双触发没了；单帧也能过且解码更稳，所以漏检大幅下降；输入链路和判决链路都更稳健，所以音量拉高后也不容易一碰就坏。

# 一、双触发问题（最明显的改进）

## 现象

* 播 1 次 → 触发 2 次甚至更多
* 尤其在有回声/混响时更严重

---

## 关键改动

### ① 只处理“新 chirp 峰”

```
state.last_peak_absif abs_peak <= state.last_peak_abs:    continue
```

### ② 触发后清空 ring

```
clear_ring_after_accept = True
```

### ③ 触发后 suppress 一段时间

```
fast_post_accept_s = 1.25slow_post_accept_s = 2.0stop_post_accept_s = 3.0
```

---

## 原理

以前的逻辑是：

> 每一帧 ring 都重新扫描 → 同一个 chirp 会被多次看到

现在变成：

1. **记住已经扫到哪个时间点（stateful）**
2. **旧的峰不再处理**
3. **触发后直接把历史波形清掉**
4. **再加一段“冷却时间”**

---

## 为什么有效

它同时堵住了三种重复来源：

| 来源 | 如何被消除 |
| :--- | :--- |
| 同一帧被重复扫描 | last_peak_abs |
| ring 里残留旧波形 | clear_ring |
| 回声 / 尾波再次触发 | suppress window |

👉 所以“双触发”几乎彻底消失

---

# 二、漏检问题（你说“几乎没有漏检”）

## 现象

* 明明播了，但完全没触发
* 尤其 fast 更容易漏

---

## 关键改动

### ① 接受条件从“多帧投票” → “单帧也可”

```

```

---

### ② Fast 引入“结构约束配对”

```

```

---

### ③ chirp refine + 多相位解码

```
refine_window_sphase_div
```

---

## 原理

### 旧系统的问题

> 必须 2 帧都成功 → 才 accept

但现实中：

* 第二帧经常被噪声/混响破坏
* 或 bit 边界稍微错位

👉 导致整包失败

---

### 新系统做法

#### 第一层：降低门槛

* 一帧 OK 就可以触发

#### 第二层：避免误判

* 如果有第二帧，必须符合：

* 正确时间间隔
* cmd 一致

#### 第三层：提高单帧成功率

* chirp 起点 refine
* 多 phase 尝试

---

## 为什么有效

👉 本质变化：

| 旧 | 新 |
| :--- | :--- |
| “必须完美” | “容忍局部失败” |
| 强依赖重复帧 | 单帧也能成立 |
| bit 边界敏感 | 多 phase 抵抗 |

👉 所以漏检大幅下降

# 三、音量 / 削波问题（你感知很明显）

## 现象

* 以前音量大 → 很容易失效
* 现在音量很大也还能识别

---

## 关键改动

### ① mic\_level 监控

```
clip_warn_level = 0.90
```

### ② 解码更鲁棒

* chirp 匹配用归一化相关
* 多 phase 解码
* 单帧 accept

### ③ 输入链路修复（最关键）

```
input_sr = 48000→ resample → 16000
```

---

## 原理

削波本质影响：

* 波形被“截平”
* 高频谐波出现
* bit 能量被污染

但你现在系统：

### 抵抗削波的机制

1. **chirp 是相关匹配**

* 对形状更敏感
* 对幅度不敏感
2. **FSK 用能量判决**

* 不是精确波形匹配
* 对失真容忍更高
3. **多 phase 尝试**

* 即使边界乱了也能对齐
4. **单帧 accept**

* 不要求两帧都完美

---

## 真正的关键点（很多人会忽略）

👉 **不是“削波没了”，而是“系统更抗削波了”**

# 四、Fast 双触发 / 误配对问题

* 两个不同 chirp 被错误当成一组
* 或同一帧被当成两帧

---

## 关键改动

### Fast repeat 结构约束

```
expected_offset = silence + chirp + payload + gap
```

允许误差：

```

```

---

## 原理

Fast 是这种结构：

```

```

所以第二个 chirp **必须出现在“正确时间位置”**

---

## 为什么有效

👉 防止两种错误：

| 错误 | 现在结果 |
| :--- | :--- |
| 噪声峰当第二帧 | 被拒 |
| 回声当第二帧 | 被拒 |
| 随机 chirp 配对 | 被拒 |

# 六、整体架构变化（最重要总结）

你这次其实不是“调参数”，而是完成了一个**架构升级**：

---

## 旧版本（脆弱系统）

```
audio → ring → 每帧重扫 → decode → 强投票 → trigger
```

问题：

* 重复触发
* 漏检严重
* 对噪声极敏感

---

## 新版本（工程化系统）

```
audio(48k)   ↓resample → 16k   ↓stateful ring   ↓只扫新峰   ↓chirp refine   ↓multi-phase decode   ↓mode-aware accept（fast/slow）   ↓post-accept 控制（clear + suppress）
```

---

你这次成功的本质是：

> **从“每帧重新猜一遍” → 变成“有记忆、有结构约束、有容错的流式检测系统”**
